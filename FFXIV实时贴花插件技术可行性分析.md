# FFXIV 实时贴花绘制插件：技术可行性分析报告

## 核心结论

**构建一个集成 Penumbra 生态的游戏内贴花投影与贴图绘制工具，在技术上完全可行。** 所有关键子系统——网格数据访问、运行时贴图替换、GPU 贴图操作、Mod 导出——在 FFXIV Mod 生态中都有可运行的先例。核心管线将结合 Meddle 风格的网格提取、DX11 计算着色器合成、Penumbra 的临时修改 IPC 实现实时预览，以及 BCnEncoder.NET 完成最终 .tex 烘焙。

目前没有任何现有工具提供这一能力。最接近的类比是 Skyrim 的 RaceMenu 叠加层系统和离线工具 Loose Texture Compiler，FFXIV Mod 工作流中存在显著空白，这个插件将填补这一空缺。

---

## 一、FFXIV 的贴图架构：约束与机会并存

FFXIV 的角色渲染依赖于由专用着色器包管理的多通道贴图系统。理解这些内部细节对于正确的贴花合成至关重要。

### 1.1 皮肤贴图结构

皮肤使用 `skin.shpk` 着色器，包含三张主要贴图：

| 贴图类型 | 通道分配 | 说明 |
|---------|---------|------|
| **法线贴图 (_n)** | RG = 切线空间法线（OpenGL Y+ 约定），B = 皮肤颜色/脱色影响，A = 上下文相关遮罩 | A 通道在身体上为湿度，在面部为唇部遮罩，在硝子族为毛发遮罩 |
| **漫反射贴图 (_d)** | 标准 RGBA 颜色 | 基础颜色信息 |
| **遮罩贴图 (_s/_m)** | R = 镜面反射强度，G = 粗糙度，B = 次表面散射厚度 | 控制材质属性 |

原版身体贴图分辨率为 **1024×1024**，面部为 **256×512**。Mod 社区常用的升级版达到 2048 或 4096。所有尺寸必须为 2 的幂次但不必正方形。

### 1.2 装备贴图结构

装备使用 `character.shpk` 着色器，采用根本不同的方案。Dawntrail（7.0）引入了专用 **索引贴图（_id）**：R 通道选择色组行对（1-16），G 通道控制 A/B 行混合，取代了 7.0 之前在法线贴图 Alpha 通道中编码行选择的旧系统。法线贴图在 RG 通道携带标准切线空间法线，B 通道为不透明度。遮罩贴图的 R 通道存储镜面反射，G 通道存储粗糙度，B 通道存储环境光遮蔽。

值得注意的是，装备模型支持 **UV Channel 2** 用于部队徽章贴花——这个模式与贴花投影直接相关。

### 1.3 贴图路径约定

贴图路径遵循严格约定：

```
chara/human/c{种族代码}/obj/body/b{变体}/texture/--c{种族代码}b{变体}_{类型}.tex
```

类型后缀：`_d`（漫反射）、`_n`（法线）、`_s`/`_m`（遮罩）。材质通过完整游戏路径引用贴图，并指定着色器常量（如 `g_alphathreshold` 用于透明度，`g_ToonIndex` 用于卡通着色参数）。

### 1.4 .tex 二进制格式

.tex 文件使用 **80 字节头**，包含属性标志、TextureFormat 枚举、尺寸、Mip 级别数（1-13 级）和 13 个表面偏移条目。各压缩格式对应特定用途：

| 压缩格式 | 用途 | 说明 |
|---------|------|------|
| **BC5** | 法线贴图 | 双通道、高质量 |
| **BC1** | 不透明漫反射 | 最小文件体积 |
| **BC3** | 带 Alpha 的漫反射 | 支持透明度 |
| **BC7** | Dawntrail 高质量遮罩/漫反射 | 7.0 新增 |

色组嵌入在 `.mtrl` 文件中，使用 **R16G16B16A16 半精度浮点**格式。贴花插件必须输出与每个通道类型严格匹配格式的贴图。

### 1.5 身体 Mod 的 UV 布局：核心兼容性挑战

FFXIV 身体 Mod 生态在 UV 空间上存在不兼容的布局分裂：

| 身体 Mod | UV 特点 | 材质槽位 |
|---------|---------|---------|
| **Vanilla/Gen2** | 对称镜像 UV | `_a` |
| **Gen3/Tight&Firm** | 完全非对称 UV（左右独立细节） | `_b` |
| **Bibo+** | 镜像但非对称布局 | `_bibo` |
| **TBSE** | 继承 The Body 2.0/2.5 UV 布局，使用硝子族着色器变体链接体毛颜色 | `_b` |

这些 Mod 同时替换网格和 UV 映射——为某一布局烘焙的贴图在另一布局上会产生错乱结果。**投影工具必须检测当前激活的身体 Mod**（通过 Penumbra IPC 路径解析查询）或让用户手动指定，然后烘焙到正确的 UV 空间。

## 二、投影管线：从 3D 空间到 UV 空间像素

核心技术挑战是将用户在 3D 屏幕空间中定位的贴花转换为正确 UV 映射的贴图数据。两种互补的方法均已在游戏引擎和 DCC 工具开发中得到充分验证。

### 2.1 贴图空间渲染（Texture-Space Rendering）——最优雅的 GPU 方案

这一技术用 UV 坐标作为顶点位置来渲染目标网格，同时将 3D 顶点位置通过贴花的投影矩阵进行投影：

**工作原理：**
1. 顶点着色器将 UV 重映射到裁剪空间（`vertex.xy = uv * 2 - 1`）
2. 从原始 3D 位置计算投影贴花坐标
3. 片段着色器对投影坐标进行范围检查（是否在贴花的单位盒内）
4. 采样贴花贴图

**优势：** 自然处理多个 UV 岛——GPU 独立光栅化所有三角形——并以全 GPU 速度运行。只需网格的顶点缓冲（位置 + UV）和贴花的视图-投影矩阵。

### 2.2 反向投影（Reverse Projection）——计算着色器方案

预计算 UV 空间中的位置图和法线图（每个网格一次性成本），然后逐像素迭代：将每个像素的 3D 位置变换到贴花空间、范围检查、采样。这是 O(width × height) 的方法，用计算着色器实现更简单，避免了光栅化的复杂性。对于 1024×1024 目标，处理约 100 万个像素——对现代 GPU 来说微不足道。

### 2.3 法线贴图混合：必须使用 Reoriented Normal Mapping

将贴花法线正确合成到基础法线上对视觉质量至关重要。**Reoriented Normal Mapping（RNM）** 方法提供最准确的结果——计算从参考法线 [0,0,1] 到基础法线的最短弧旋转，然后将此旋转应用于细节法线。优化后的 HLSL 非常简洁：

```hlsl
float3 t = baseNormal.xyz * float3( 2,  2, 2) + float3(-1, -1,  0);
float3 u = detailNormal.xyz * float3(-2, -2, 2) + float3( 1,  1, -1);
float3 r = t * dot(t, u) / t.z - u;
```

成本仅约 5-6 个 ALU 指令，完整保留两个法线的强度。更简单的 **UDN 混合**（`r = normalize(n1.xy + n2.xy, n1.z)`）节省几个指令但在超过 45° 时会压平法线——对微妙的纹身可接受，但对深度伤疤或浮雕徽章不够用。

对于 FFXIV 的 BC5 法线贴图（只存储 RG，Z 通过重建获得），插件必须在混合前重建 Z，混合后重新推导 Z。

### 2.4 UV 接缝处理：迭代膨胀

当贴花跨越 UV 岛边界时，岛间间隙的像素没有数据，导致 Mipmap 时出现可见接缝。标准解决方案是**贴图膨胀（Texture Dilation）**：每个空像素采用最近已填充邻居的颜色。推荐的填充像素数随分辨率缩放：**1K 时 8 像素，2K 时 16 像素，4K 时 32 像素**。Jump Flood Algorithm 变体可在 O(log n) 次传递中实现大范围填充。用约 10 次 8 连通泛洪填充计算着色器传递即可实现"无限膨胀"（填充所有间隙空间），成本可忽略不计。

## 三、Dalamud 提供了所有必需的钩子

Dalamud 插件框架及其生态提供了充分的——虽然有时脆弱的——对 FFXIV 渲染内部的访问能力。

### 3.1 网格数据完全可访问

**Meddle**（PassiveModding/Meddle）证明了从运行中的游戏完整提取网格数据是可行的。它通过 **FFXIVClientStructs** 读取 `CharacterBase` 结构，提取顶点、法线、UV、索引、骨骼权重和形态键。FFXIV 在 CPU 端维护模型数据用于角色渲染管理，无需 GPU 顶点缓冲回读。

**Lumina** 库提供互补的离线解析能力，可以从游戏的 SqPack 归档中解析 `.mdl` 文件，暴露顶点声明（位置、法线、UV、颜色、混合权重）、网格结构和材质引用。

两种方法结合使用：实时从游戏内存读取（获取精确的当前角色状态）或从游戏文件解析（用于可预测的离线处理）。

### 3.2 运行时贴图替换：成熟的 API 路径

最实用的预览管线使用 **Penumbra 的 IPC**：

1. `CreateTemporaryCollection()` —— 创建仅会话有效的临时集合
2. `AddTemporaryMod()` —— 注册文件重定向，将游戏贴图路径指向临时文件
3. 触发角色重绘 —— 显示更新后的贴图

这完全避免了脆弱的渲染管线钩子——游戏只是加载了一个不同的文件。

对于更紧密的集成，**Dalamud 的 ITextureProvider** 可以创建带 CPU 读写访问的空 GPU 贴图（`CreateEmpty(RawImageSpecification, cpuRead: true, cpuWrite: true)`），并将 `IDalamudTextureWrap` 实例转换为游戏 `Texture` 内核对象。但直接注入游戏材质系统需要钩住 `PSSetShaderResources`——可行但不受官方支持。

### 3.3 DirectX 11 互操作：Dalamud v14 使用 TerraFX

**SharpDX 已在 Dalamud API v14 中移除**（SharpDX 自 2019 年起已停止维护）。推荐替代方案是 **TerraFX.Interop.Windows**，提供对 DirectX 11 API 的原始 COM 风格绑定。插件可通过 Dalamud 的交换链钩子访问 `ID3D11Device` 和 `ID3D11DeviceContext`。创建计算着色器、渲染目标、暂存贴图和发起调度/绘制调用在技术上可行，但需要仔细管理状态以避免破坏游戏的渲染状态。

**自定义着色器传递**不受官方支持。游戏的 `Render::Manager::Render` 函数可通过 `IGameInteropProvider.HookFromSignature()` 进行钩住，允许在游戏渲染传递前后执行代码并获得完整的 DX11 设备访问。这很脆弱——签名会随游戏更新而失效——但对于不干扰光栅化管线的计算着色器调度来说是可行的。

## 四、现有生态的缺口与可复用的构建模块

### 4.1 FFXIV 完全没有游戏内绘制工具

当前的纹身制作工作流完全在离线进行，且相当痛苦：

```
获取目标身体 Mod 的基础贴图
    ↓
在 Photoshop/GIMP 中于扁平化 2D UV 布局上绘制
    ↓
手动在漫反射/法线/遮罩通道之间合成（每个通道有 FFXIV 特定语义）
    ↓
导出为正确格式的 .tex 文件
    ↓
打包为 Penumbra Mod（含种族特定路径变体）
```

**Tattoo Master Rig**（一个带 FFXIV 种族模型的 Blender 场景文件）提供 3D 绘制但完全在游戏外操作。**Loose Texture Compiler** 自动化了打包和身体类型转换（使用 XNormal 进行 UV 重投影）但仍然是离线批处理工具。

**Aetherment**（Sevii77/aetherment）是向运行时合成迈进最远的现有项目。用 Rust 编写，带有 C# Dalamud 包装器，宣传"高级自动贴图合成"，但文档稀少，社区采用率低于 Penumbra/LTC 生态。其架构——作为 DLL 加载的原生 Rust 核心——证明了性能关键的贴图操作可以在 Dalamud 插件模型内运行。

### 4.2 RaceMenu 和 LooksMenu 在其他引擎中验证了这一概念

**Skyrim 的 RaceMenu** 钩入引擎原生的着色层系统，可同时合成**每个身体区域 6 层以上叠加层**，每层具有独立的颜色、透明度和发光参数，可在游戏内调节。叠加层是运行时通过引擎着色器管线着色的灰度 DDS 贴图。

**Fallout 4 的 LooksMenu** 增加了 UV 变换能力——叠加层可以在身体上实时重新定位和缩放，这一功能与贴花定位直接相关。

两个系统都受益于引擎级别的叠加层支持——**FFXIV 完全缺少这一能力**。FFXIV 的皮肤着色器没有原生的叠加/着色层系统，每个材质只读取单组贴图。任何叠加合成都必须在着色器看到贴图之前，在贴图数据层面完成——**计算着色器预合成是唯一可行的方案**。

### 4.3 基于色组的替代方案存在但有限

社区探索了通过 FFXIV 的**色组系统**编码纹身（"ALum"方法）。通过法线贴图 Alpha 通道将纹身区域分配到特定色组行，纹身可以通过游戏标准染色界面变色。这个巧妙的方案避免了贴图合成，但限制纹身只能使用色组行定义的平面颜色——没有照片级细节、渐变或复杂着色。它与贴图空间投影绘制互补而非替代。

## 五、可构建的架构设计

以下架构综合了所有调研结论，形成一个具体的、可实现的系统。

### 5.1 实时预览：计算着色器合成管线

预览管线在每次贴花状态变化时串联三个计算着色器传递：

**传递 1 — 投影传递：**
对目标贴图的每个像素，着色器从预计算的位置图和法线图中读取数据（角色加载时从网格数据一次性生成）。将每个像素的 3D 位置变换到贴花投影器空间，范围检查（是否在贴花体积内），根据投影器方向进行背面剔除，采样贴花贴图，写入合成缓冲。在 **1024×1024** 分辨率下，以 `[numthreads(8,8,1)]` 调度约 131K 线程组——过去十年内的任何 GPU 上都在 1ms 以内。

**传递 2 — 合成传递：**
使用可配置的混合模式将贴花贡献与基础贴图混合。漫反射使用 Porter-Duff source-over 标准 Alpha 混合，艺术效果使用 multiply/overlay/soft-light。法线贴图应用 RNM 混合。遮罩贴图按通道使用适当语义合成（贴花粗糙度混合、镜面反射修改）。

**传递 3 — 膨胀传递：**
运行 8-16 次最近邻泛洪填充以填充 UV 岛间隙。

合成后的贴图写入临时文件。通过 Penumbra 的 `AddTemporaryMod` IPC 将游戏路径重定向到此文件。触发角色重绘（通过 Penumbra 或 Glamourer IPC）显示结果。**整个过程不需要任何渲染管线钩子——游戏只是加载了一个不同的文件。**

### 5.2 网格数据获取与位置图生成

角色加载时（或用户启动绘制时），插件使用 **Meddle 方法**读取目标模型的顶点数据——通过 FFXIVClientStructs 遍历 `CharacterBase` → `Model`，提取顶点位置、法线和 UV 坐标。对于装备或内存访问不可靠时，可通过 Lumina 解析 `.mdl` 文件。

从网格数据中，插件将所有三角形光栅化到 UV 空间（以 UV 作为位置，插值 3D 位置和法线），生成投影计算着色器消费的**位置图**和**法线图**。这个光栅化是一次性成本——典型身体网格约 **5-10ms**。

### 5.3 烘焙管线：导出持久化 Penumbra Mod

用户最终确认贴花位置后，烘焙管线执行以下步骤：

```
1. 最终合成（目标分辨率，可选 2× 超采样 + Lanczos 缩放以获得抗锯齿贴花边缘）
    ↓
2. 完全膨胀（无限——填充所有间隙空间）
    ↓
3. Mipmap 生成（ID3D11DeviceContext::GenerateMips()）
    ↓
4. GPU 回读（通过暂存贴图：Usage.Staging + CpuAccessFlags.Read，
   单次 CopyResource + Map 调用，1K 贴图约 2ms 延迟）
    ↓
5. BC 压缩（BCnEncoder.NET，纯 C#，NuGet 可用）：
   - BC7 用于漫反射和遮罩贴图
   - BC5 用于法线贴图
   - "Balanced"质量级别，1024×1024 BC7 编码约 200-500ms
    ↓
6. 写入 FFXIV .tex 文件（匹配 Lumina TexFile 头格式：
   80 字节头 + 正确的 TextureFormat 枚举 + 尺寸 + Mip 数 + 表面偏移 + 压缩 Mip 数据）
    ↓
7. 生成 Penumbra Mod：
   - meta.json（FileVersion 3，Mod 名称/作者/标签）
   - default_mod.json（游戏贴图路径映射到本地文件）
   - 可选 group_001_BodyType.json（Vanilla/Bibo+/Gen3/TBSE 变体的单选选项）
```

对于多身体 Mod 支持，插件可在单次操作中烘焙到所有支持的 UV 布局——每个布局只是使用从对应身体 Mod 的网格数据生成的不同位置/法线图（通过 Lumina 从 Mod 的 .mdl 文件获取）。

## 六、关键技术风险与缓解策略

### 6.1 签名断裂（最高风险）

钩入 `Render::Manager` 或 `CharacterBase` 结构的签名会随每次游戏更新而失效。

**缓解策略：** 核心管线（计算着色器合成 → 文件写入 → Penumbra IPC）完全不需要渲染钩子。只有网格数据提取路径依赖 FFXIVClientStructs，而该项目有快速的社区更新节奏（通常在补丁后数小时内）。回退到基于 Lumina 的 .mdl 文件解析可消除即使这一依赖（用于离线烘焙）。

### 6.2 性能

整个预览管线——位置图查找、贴花投影、合成、膨胀、Penumbra 重载——总计应在 **50ms 以内**完成，主要瓶颈在文件 I/O 和角色重绘而非 GPU 工作。对于交互式定位（移动/旋转贴花），预览可以 5-10 Hz 更新而无可感知延迟。

### 6.3 贴图分辨率限制

在原版 1024×1024 身体贴图上，覆盖约 15% UV 空间的胸部纹身获得大约 **150×150 像素**的有效分辨率——对大胆设计足够，对精细线条勉强。需要高细节的用户必须使用升级的基础贴图（2K 或 4K），这在 Mod 社区已是标准做法。

### 6.4 贴花定位 UI（最高复杂度风险）

渲染与游戏摄像机和角色对齐的 3D Gizmo 或投影预览需要：
- **方案 A**：ImGui 叠加层 + 手动世界到屏幕投影（可行但深度感知不精确）
- **方案 B**：钩入渲染管线实现正确的深度测试可视化（脆弱）

**建议的降低风险策略：** 从更简单的方案开始——在单独的视口中使用 Meddle 导出的网格数据的轨道摄像机预览，而不是叠加在实时游戏上。这样可以在证明核心合成和烘焙管线的同时降低初始复杂度。

## 七、开发路线图建议

### 第一阶段：核心验证（MVP）

- 通过 Lumina 解析 .mdl 文件获取网格数据（避免 FFXIVClientStructs 依赖）
- 实现 CPU 端的反向投影（暂不用计算着色器）
- 仅支持漫反射通道合成
- 输出 .tex 文件到 Penumbra Mod 目录
- 仅支持 Vanilla 身体 UV
- ImGui 窗口中的 2D UV 空间贴花定位

### 第二阶段：GPU 加速与多通道

- 迁移到 DX11 计算着色器管线
- 添加法线贴图（RNM 混合）和遮罩贴图通道
- 通过 Penumbra 临时 Mod IPC 实现实时预览
- 支持多个贴花层叠加

### 第三阶段：完整体验

- 3D 视口贴花定位（使用 Meddle 提取的实时网格）
- 多身体 Mod UV 布局支持（Bibo+/Gen3/TBSE 自动检测）
- 装备/武器贴花支持
- 完整的 .pmp Mod 导出（含选项组）
- 贴花库管理（保存/加载贴花预设）

### 第四阶段：高级功能

- 游戏内投影预览叠加（需要渲染钩子）
- 曲面自适应贴花变形（利用法线数据调整贴花以匹配表面曲率）
- 多分辨率支持（自动检测基础贴图分辨率并匹配）
- 导出为 Heliosphere 可分发的 Mod 包

## 八、结论

实时贴花绘制插件的技术基础分散在成熟但碎片化的组件中。**Meddle 证明了网格访问可行，Penumbra 的临时 Mod IPC 提供了干净的无钩子预览机制，DX11 计算着色器以极低成本处理实时合成，BCnEncoder.NET 用纯 C# 解决了 BC7 压缩，贴图空间渲染技术有充分文档和现成实现。**

这个项目真正的创新贡献在于**将这些碎片连接成统一的游戏内工作流**——这是任何游戏 Mod 生态中目前都没有实现的集成级别。最接近的先例（RaceMenu 叠加层、Substance Painter 投影）要么依赖引擎原生支持，要么作为独立应用运行。一个能读取实时网格数据、通过计算着色器合成、通过 Penumbra Mod 系统预览、并导出持久化 Mod 的游戏内投影绘制系统，将是**同类首创的工具**。
